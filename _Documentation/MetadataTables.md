# ERP Platform — Universal Record Services

---

## 1. The Core Idea

Every ERP entity — AP Invoice, AR Customer, PO, GL Batch — carries a `RecordID` (`uniqueidentifier`). That single key unlocks a shared platform layer that every module gets for free.

```
Any ERP Entity  (APInvoice, ARCustomer, PO, GLBatch, Project…)
        │
        │  RecordID  ◄── registered in SYRecordType
        │
        ├──► Tags               (SMTag + SYRecordTag)
        ├──► Global Search      (SYRecordSearchIndex)
        ├──► File Attachments   (SMFile + SMFileRevision + SYRecordAttachment)
        ├──► Comments           (SYRecordComment)
        └──► Links to Records   (SYRecordReference)
```

## 2. Tags — `SMTag` + `SYRecordTag`

### Overview
A two-table design: a shared tag dictionary and a junction that assigns tags to records.
Tags help to group related records and review/process them together.

### Functionality
* Tags are defined once in `SMTag` and reused across any entity type.
* `Scope` (char 64) scopes which entity types a tag applies to — e.g., `"Account"`, `"Documents"`, `"Warehouse"`. The UI filters the available tag list by the current entity's type. The scope is defined by the developers and cannot be changed.
* `Color` (hex) is stored in the database — badge appearance is driven by data, not frontend constants.
* `SYRecordTag` is a junction of `RecordID + TagID`. Multiple tags per record; the same tag across many records.

## 3. Global Search — `SYRecordSearchIndex`

### Overview
A pre-built, locale-aware search index written on every record save. User search queries hit this table only — source tables are never touched at query time.

### Functionality
* One row per record per locale, enabling locale-specific full-text content and result display.
* `Content` is a denormalized concatenation of all searchable text (vendor name, reference number, description, line items…). A SQL Server Full-Text index on `Content` powers keyword search.
* Faceting columns support filtered search without joins to source tables:

  | Column | Example filter |
  |---|---|
  | `ModuleCode` | "show AP results only" |
  | `Status` | "show Open records" |
  | `Date` | "last 30 days" |
  | `Amount` + `CurrencyID` | "invoices over $10,000 in EUR" |
  | `OwnerID` | "my records" |

* `Title` and `Description` are pre-computed for display.

## 4. File Attachments

### Overview
A three-layer design separating upload validation, file storage with revision history, and record association.

### Functionality
* **`SMFileType`** — upload whitelist keyed by extension. `Allowed` bit gates every upload before anything is written. `IsImage` enables inline preview.
* **`SMFile`** — the file header: name and a pointer to the current revision (`LastRevisionID`).
* **`SMFileRevision`** — append-only. Every upload of a new version adds a row; no row is ever updated. Full revision history is always preserved.
  * `BlobHandler = NULL` → bytes stored inline in `Data`.
  * `BlobHandler ≠ NULL` → bytes live in external storage (Azure Blob, S3); the DB holds only the reference. Small files can be inline, large files external — hybrid by design.
* **`SYRecordAttachment`** — links a record to a file (`RecordID + FileID`). The same `FileID` can be attached to multiple records; no file duplication.

## 5. Comments — `SYRecordComment`

### Overview
User notes and system-generated audit messages in one chronological stream per record.

### Functionality
* `Type = U` — written by a person; `Type = S` — generated by workflow or automation. Both types live in the same table and are rendered as a single activity feed. No separate "notes" vs "audit log" panel.
* `RefActivityID` traces a system comment back to the specific workflow event that triggered it — auditors can navigate from a comment to the approval step, approver, and timestamp.

## 6. Record Links — `SYRecordReference`

### Overview
A universal two-GUID junction that links any record to any other record, regardless of entity type or module.

### Functionality
* Two columns only: `SourceRecordID` + `DestRecordID`.
* Type column allows to categorize links.
* Clustered index on `(Source, Dest)` and a unique index on `(Dest, Source)` — both traversal directions are equally fast.
* Adding a new relationship type between any two entities requires only application logic.
* Supports user-defined and partner-defined links without schema modifications.
